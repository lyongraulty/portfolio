## Content workflow

This folder is the handoff point between your asset storage and the website.

The goal is simple:

- Bunny stores the actual media files.
- `content/projects.json` is the project manifest the site reads ŌĆö edit this directly.
- `content/bunny-media.json` maps Bunny files to rendering roles.
- `content/site-data.json` is the local-first runtime source for pages, tokens, and fonts.

### Page ID convention

Page IDs are integers. **Page 0 is the reel.** Projects start at page 1 and increment from there. The code enforces this via `REEL_PAGE_ID = 0` in `src/lib/pageData.ts` ŌĆö change it there if the convention ever shifts, and all filters update automatically. Never hardcode the comparison in components.

## Recommended Bunny structure

Use one folder per project with predictable file names. The folder name must match the `slug` in `projects.json`.

```text
/projects
  /sellout
    sellout_card.mp4
    still-01.png
    still-02.png
  /drumwave
    drumwave_card.mp4
    still-01.png
  /reebok-sustainability
    reebok-sustainability_card.mp4
```

The `*_card.<ext>` filename is the convention the sync script uses to identify the card media. Other files are inventoried but not auto-wired unless `BUNNY_AUTO_BLOCKS=true`.

## Manifest rules

In `content/projects.json`:

- `slug` should match the Bunny folder name.
- `media.src` can be:
  - a full URL, or
  - a relative path like `projects/drumwave/cover.mp4`
- If `media.src` is relative, the app prefixes it with `BUNNY_MEDIA_BASE_URL`.

Example:

```json
{
  "slug": "drumwave",
  "title": "DRUMWAVE",
  "media": {
    "type": "video",
    "src": "projects/drumwave/cover.mp4"
  }
}
```

## Practical workflow

1. Upload media files to Bunny using the folder/filename convention above.
2. Run `npm run sync:bunny-media` ŌĆö auto-indexes Bunny folders and writes `content/bunny-media.json`.
3. Run `npm run sync:local-pages` ŌĆö regenerates project pages (page Ōēź 2) from `content/projects.json` and `content/bunny-media.json`.
4. Edit copy, order, year, and category directly in `content/projects.json`.
5. Optionally adjust block layout/roles in `content/bunny-media.json` (script preserves manual edits on future runs).
6. Edit `content/site-data.json` for the reel page (page 1), tokens, and fonts.
7. Set `NEXT_PUBLIC_MEDIA_BASE_URL` in `.env.local` to your Bunny pull zone domain if using relative asset paths.
8. Run `npm run validate:content` before deploy to catch schema issues early.

Optional runtime flags (`.env.local`):
- `CONTENT_SOURCE=remote` ŌĆö prioritize remote Apps Script at runtime.
- `CONTENT_ENABLE_REMOTE_FALLBACK=true` ŌĆö use remote only when local data is unavailable.

## Bunny auto-index setup

`npm run sync:bunny-media` requires:

- `BUNNY_STORAGE_ZONE`
- `BUNNY_STORAGE_API_KEY`

Optional:

- `BUNNY_STORAGE_HOST` (default `storage.bunnycdn.com`)
- `BUNNY_MEDIA_ROOT` (default `projects`)
- `BUNNY_PROJECT_SLUGS` (comma-separated subset, e.g. `colossal,drumwave`)
- `BUNNY_AUTO_BLOCKS` (default off; set `true` to auto-generate blocks from filenames)

Current convention:

- Any filename matching `*_card.<ext>` is treated as the project card media.
- Other files are inventoried under `files` and are not auto-wired into blocks unless `BUNNY_AUTO_BLOCKS=true`.

This script writes relative paths (for example `projects/colossal/shot-01.mp4`) and adds a `files` inventory snapshot per project so URL lookup is easy without Sheets.

## Bunny mapping file

`content/bunny-media.json` is generated by `npm run sync:bunny-media`. You can also create it manually.

This file is the bridge between folders and rendering intent:

- `slug` matches the sheet row and project folder
- `media.card` is the preview asset for the project grid
- `media.blocks` defines the project detail page media and layout

This keeps "where the file lives" separate from "how the site should use it."

## projects.json field reference

Each entry in `content/projects.json` supports:

- `slug` ŌĆö must match the Bunny folder name
- `page` ŌĆö integer Ōēź 2 (page 1 is reserved for the reel)
- `title`
- `year`
- `category`
- `summary` ŌĆö short description shown on card and in modal
- `detail` ŌĆö longer body copy for the project page
- `button_text` ŌĆö CTA label (defaults to "View Project")
- `media.type` ŌĆö `"video"` or `"image"`
- `media.src` ŌĆö relative path (`projects/slug/slug_card.mp4`) or full URL

## Why this helps

- One place to organize project metadata (`projects.json`)
- Bunny paths become predictable instead of ad hoc
- `bunny-media.json` keeps "where the file lives" separate from "how the site uses it"
- The validated JSON is the clean contract your site renders
